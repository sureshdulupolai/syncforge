Step 1

Installation

Install the SyncForge SDK with pip. Works in any Python 3.8+ environment — Django, FastAPI, Flask, or a standalone script.

Terminal 
# Install via pip
pip install syncforge

# Or for Node.js
npm install syncforge
The SDK uses only the standard library + requests. No heavy dependencies. Works offline — it only contacts SyncForge when you explicitly call .refresh().
Step 2

Quick Start

Three steps: get your API key from the dashboard → create a syncforge.py file → call .refresh() after any database write.

Python — any framework 
from syncforge import SyncForge

# Initialize once — like Flask or FastAPI app
app = SyncForge(api_key='sf_live_YOUR_KEY')

# After any database write, call refresh()
app.refresh('products')   # table name you registered in the dashboard

The syncforge.py Pattern

Just like Celery has a celery.py file in your project root, SyncForge recommends a dedicated syncforge.py file per project. This keeps your API key in one place and makes it easy to import anywhere.

myproject/  →  syncforge.py   (same level as manage.py)
syncforge.py — project root 
"""
syncforge.py
Place this file in your project root (same folder as manage.py / main.py).
Import `sf` anywhere in your project to trigger syncs.
"""
import os
from syncforge import SyncForge

# Read from environment variable (recommended) or hardcode for development
sf = SyncForge(
    api_key=os.environ.get('SYNCFORGE_API_KEY', 'sf_live_YOUR_KEY_HERE')
)

Then import and use it anywhere in your codebase:

views.py / routes.py / any file 
from syncforge import sf   # import the shared instance

def create_product(request):
    Product.objects.create(name='New Item', price=99.99)

    # One line — notify all connected clients
    sf.refresh('products')

    return JsonResponse({'status': 'created'})

API Keys

Each project in your SyncForge dashboard gets its own set of API keys. Keys are scoped to a single project — using one key cannot affect another project's tables. You can create up to 5 active keys per project.

Key format
sf_live_a1b2c3d4e5f6...

48 hex characters after the prefix. Cryptographically random.

How to pass the key
X-API-Key: sf_live_...

Send as an HTTP header in every request to the SyncForge API.

Never commit API keys to Git. Use environment variables: os.environ.get('SYNCFORGE_API_KEY')

Sync Modes

Each table in your project has its own sync mode. Choose the right mode based on how often the data changes.

manual
Manual Mode

Zero DB calls until you call sf.refresh('table') or click Refresh in the dashboard. Perfect for config tables, feature flags, pricing data.

schedule
Schedule Mode — 5m / 1h / 1d / 30d

SyncForge auto-refreshes on a timer. No polling from clients. Your DB only gets hit on the scheduled interval — not once per client.

event
Event Mode — INSERT / UPDATE / DELETE

Call sf.refresh() from your write handlers. SyncForge broadcasts the delta immediately. Best for high-velocity tables.

hybrid
Hybrid Mode

Combines Event + 24h scheduled verification. Use when you need instant updates but also want a safety net to catch missed events.

Tables

A "table" in SyncForge maps to a table (or collection) in your database. You register it in the dashboard and SyncForge tracks sync stats per table.

Duplicate Table Prevention

SyncForge enforces unique table names per project at the database level (unique_together constraint). If you try to add the same table twice — during testing or otherwise — the API returns a clear error instead of creating a duplicate.

Response — table already exists 
{
  "error": "Table \"products\" already exists in this project."
}
Safe to run sf.refresh('products') multiple times — it never creates duplicates. Only the dashboard "Add Table" form creates table records.

Triggering a Sync

Call sf.refresh('table_name') immediately after any database write that should propagate to clients.

Python SDK 
# Single table
sf.refresh('products')

# Multiple tables at once
sf.refresh('products', 'categories', 'inventory')

# With result check
result = sf.refresh('products')
if result.ok:
    print(f"Sync triggered — {result.calls_saved} DB calls saved so far")

Fetch & Update Flow

Here's the complete lifecycle when data changes — from your backend to all connected client devices:

1
Your backend writes to DB — e.g. Product.objects.create(...)
2
You call sf.refresh('products') — sends POST /api/v1/sync/products/
3
SyncForge invalidates its cache for the products table and logs the event
4
Next client request gets fresh delta data — zero unnecessary polling

The raw HTTP call (what the SDK does under the hood):

Raw HTTP / curl 
curl -X POST https://syncforge.dev/api/v1/sync/products/ \
  -H "X-API-Key: sf_live_YOUR_KEY"

# Response
{
  "status": "ok",
  "message": "Sync triggered for table `products` in project `MyApp`.",
  "table": "products",
  "sync_mode": "Event — On INSERT / UPDATE / DELETE",
  "database_calls_saved": 1854211
}

Django Integration

SyncForge provides first-class support for Django with zero-code auto-syncing and a built-in Security Middleware.

1. Auto-Sync Models (Recommended)

Use the @sync_model decorator. It automatically registers the table and triggers a sync on every post_save and post_delete.

models.py 
from django.db import models
from syncforge import sf
from syncforge.django import sync_model

@sync_model(sf, sync_mode='event')
class Product(models.Model):
    name = models.CharField(max_length=100)

2. Security Middleware (WAF & Logging)

Protect your Django app from hackers. The built-in middleware blocks SQLi (`OR 1=1`), XSS (`<script>`), and Path Traversal (`../`), while logging all response times perfectly. No configuration needed!

settings.py 
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    # Add SyncForge Security Firewall
    'syncforge.middleware.SyncForgeSecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    # ...
]

3. Manual View Sync

If you prefer manual control over auto-syncing, call sf.refresh() in your views.

views.py 
from syncforge import sf

def bulk_update(request):
    Product.objects.filter(active=True).update(price=99)
    sf.refresh('products') # one sync for bulk write

FastAPI & Flask Integration

SyncForge can be easily integrated with FastAPI and Flask. Import the SyncForge client and trigger refreshes manually.

1. FastAPI

main.py 
import os
from fastapi import FastAPI
from syncforge import SyncForge

sf = SyncForge(api_key=os.environ.get('SYNCFORGE_API_KEY'))
app = FastAPI()

@app.post("/api/products/")
async def create_product(name: str):
    # 1. Database operation
    db.execute("INSERT INTO products (name) VALUES (?)", (name,))
    
    # 2. Trigger sync
    sf.refresh('products')
    return {"status": "success"}

2. Flask

app.py 
from flask import Flask, jsonify
from syncforge import SyncForge

app = Flask(__name__)
sf = SyncForge(api_key='sf_live_...')

@app.route("/api/orders/", methods=['POST'])
def create_order():
    # Database op...
    sf.refresh('orders')
    return jsonify(success=True)

REST API Reference

Base URL: https://syncforge.dev/api/

POST /api/v1/sync/<table_name>/
Triggers a manual sync for the specified table. Requires X-API-Key header.

Headers

X-API-Key: sf_live_...

Returns

{"status": "ok", ...}
GET /api/v1/project/
Returns project details and all registered tables for the API key used.
GET /api/v1/tables/
Lists all tables for the project with sync mode, row count, and calls saved stats.
GET /api/v1/health/
Public health check — no auth required. Returns {"status": "ok"}
