Deployment

Preparing your app for production with SyncForge.

1. Environment Variables

Never hardcode your API key.

Always load it from the environment. Ensure .env is in your .gitignore file.

2. Production Client Configuration

syncforge.py 
import os
from syncforge import SyncForge

sf = SyncForge(
    api_key=os.environ['SYNCFORGE_API_KEY'],
    silent=True,       # CRITICAL: Prevents your app from crashing if SyncForge API is down
    async_mode=True,   # Fire-and-forget background threads for maximum performance
)

3. Multi-Worker Deployments (Gunicorn / uWSGI)

If you deploy your app with multiple worker processes (e.g., gunicorn -w 4), you MUST use a shared cache backend like Redis or Memcached.

Why LocMemCache fails in production

Django's default LocMemCache stores data in the local memory of a single Python process. If Worker 1 handles a DB write and calls sf.refresh(), it clears its own cache. Workers 2, 3, and 4 do NOT know about the change and will continue serving stale data.

The Fix

Configure Django to use Redis. When Worker 1 clears a cache key, it deletes it from the central Redis server, so all workers instantly get the cache miss on the next request.
