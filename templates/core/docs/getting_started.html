Overview

Getting Started

Install the SDK, create a project, get your API key, and make your first sync call — all in under 5 minutes.

1 Install the SDK

terminal 
pip install syncforge

No external dependencies. Pure Python Standard Library.

2 Create a project & get your API key

  1. → Register at syncforge.dev
  2. → Create a new project in the dashboard
  3. → Copy your API key — it starts with sf_live_
  4. ⚠ The raw key is shown only once — store it in your .env immediately

3 Set your environment variable

.env 
SYNCFORGE_API_KEY=sf_live_your_key_here
Never hardcode your API key in source code. Always read it from environment variables. Your .env file must be listed in .gitignore.

2 Initialise the client

Create one SyncForge instance per project. Place it in a module-level file and import it wherever needed.

syncforge_client.py 
import os
from syncforge import SyncForge

# Initialize the SyncForge Engine
sf = SyncForge(
    api_key=os.environ.get('SYNCFORGE_API_KEY', ''),
    backend='in_memory',  # Select your storage engine
    dev_mode=True         # 🚀 Offline Development Mode (Bypasses API Key)
)
Dev Mode (Offline): Setting dev_mode=True completely bypasses network calls to the live server. You won't need an API key, and SyncForge will gracefully log simulated, structured JSON responses directly in your local terminal to help you debug without server load.

Which backend should you choose?

backend='in_memory' Recommended for Single-Node

Stores cached queries directly in your server's RAM. Zero external dependencies. Absolute lowest possible latency (sub-0.1ms). Perfect for standard deployments and VPS.

backend='redis' Recommended for Clusters

Connects to a Redis cluster to serve as a centralized cache for distributed environments. Use this if you are running multiple Docker containers across Kubernetes.

backend='django_cache'

Hijacks Django's native CACHES framework (e.g., Memcached). Useful if you already have a massive enterprise Django setup and don't want to spin up a new Redis instance.

3 Smart Cache-Aside Reading

In your view or endpoint, wrap your database fetch with sf.cache_query(). If cache_key is omitted, SyncForge securely auto-generates a unique cache key by hashing the exact query, ensuring perfect project and dataset isolation.

views.py 
from .syncforge_client import sf

# The cache_key is auto-generated based on the query if omitted.
def product_list(request):
    products = sf.cache_query(
        table_name='core_product',
        queryset=Product.objects.filter(active=True)
    )
    return render(request, 'products.html', {'products': products})

5 Signal a data change

Call sf.refresh() after every database write. SyncForge will invalidate cached data and update the server.

example.py 
from syncforge_client import sf

# After any DB write
product = Product.objects.create(name='Widget', price=9.99)

# Tell SyncForge this table changed. It will instantly invalidate cache
# across all your servers!
sf.refresh('products')

✓ Verify connectivity

python 
if sf.ping():
    print('SyncForge connected!')
else:
    print('Check your API key or network')
