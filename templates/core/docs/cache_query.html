Core Feature

cache_query()

The heart of SyncForge's read path. A cache-aside implementation with built-in stampede protection.

1. How it works

  • ✓
    Instant Reads: Returns pre-warmed data instantly from memory.
  • ✓
    Auto-Caching: On cache miss, it hits the DB and automatically saves the result.
  • ✓
    Auto-Tracking: Keys are registered internally so sf.refresh() clears them automatically.

2. Reading Data (Cache-Aside Pattern)

Follow these 3 steps to implement the cache-aside pattern. This guarantees that the database is only queried for the very first client. Every subsequent client gets data directly from the fast cache.

  1. 1
    get_table(): Instantly fetches pre-warmed data from local memory. Zero database hits.
  2. 2
    cache_query(): If memory is empty (cache miss), executes the DB query and automatically saves the result back to memory.
  3. 3
    Combine Both: Use a simple if/else block to connect them.
views.py 
from myproject.sf import sf
from .models import Product

def get_global_report(request):
    if sf.get_table('core_report'):
        # 1. Cache Hit: Data found in memory! DB query skipped entirely.
        data = sf.get_table('core_report')
    else:
        # 2. Cache Miss: Execute DB query and automatically save to memory
        data = sf.cache_query(
            table_name='core_report',
            queryset=Product.objects.all(),
            timeout=3600 # 1 hour fallback TTL
        )
        
    # 3. Return data exactly as it came from the DB
    return render(request, 'report.html', {'report': data})

3. Dynamic / Parameterized Keys

If your data depends on user input (e.g. pagination or filters), explicitly pass a cache_key to separate the data. You do not need to manually invalidate them — sf.refresh('core_report') clears all dynamic keys under that table automatically.

views.py 
def category_list(request, category_id):
    # Unique key for each category
    key = f'category_items_{category_id}'

    if sf.get_table('core_product', cache_key=key):
        data = sf.get_table('core_product', cache_key=key)
    else:
        data = sf.cache_query(
            table_name='core_product',
            cache_key=key,
            queryset=Product.objects.filter(category_id=category_id)
        )
        
    return render(request, 'list.html', {'products': data})

4. Cache Stampede Protection

The Problem:

If a popular cache key expires or is refreshed, and 500 users request that page at the exact same millisecond, all 500 requests will experience a cache miss and hit your database simultaneously, bringing it down.

The Solution (built-in):

SyncForge uses Python threading.Lock() per cache key with Double-Checked Locking. Only the first thread executes the database query. The other 499 threads wait for the lock, then instantly retrieve the result from the cache populated by the first thread. Result: exactly 1 database call.

7. Framework Agnostic Support

Note for Non-Django Users: cache_query() and get_table() integrate seamlessly with Django's native caching framework. However, if you are using FastAPI, Flask, or plain Python, SyncForge automatically falls back to an internal thread-safe memory cache. This allows you to use these powerful features natively across any tech stack!
