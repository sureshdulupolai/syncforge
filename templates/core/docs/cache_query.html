Core Feature

cache_query()

The heart of SyncForge's read path. A cache-aside implementation with built-in stampede protection.

1. How it works

  1. 1
    Check Django cache for cache_key.
  2. 2
    If cache hit: return data instantly. Fire background thread to report cache hit to SyncForge API.
  3. 3
    If cache miss: acquire lock (stampede protection), execute database queryset.
  4. 4
    Store result in cache for timeout seconds.
  5. 5
    Register cache_key under table_name registry so sf.refresh() knows what to clear later.

2. Basic usage

views.py 
from myproject.sf import sf
from .models import Product

def product_list(request):
    data = sf.cache_query(
        table_name='core_product',          # Dashboard name
        cache_key='all_active_products',   # Unique string key
        queryset=Product.objects.filter(active=True),
        timeout=3600,                      # 1 hour fallback TTL
    )
    return render(request, 'list.html', {'products': data})

3. Dynamic / parameterized keys

If your queryset depends on user input (e.g. pagination or filters), the cache_key must include those parameters.

views.py 
def category_list(request, category_id):
    # BAD: All categories will share the same cache key!
    # key = 'category_items' 

    # GOOD: Key includes the parameter
    key = f'category_items_{category_id}'

    data = sf.cache_query(
        table_name='core_product',
        cache_key=key,
        queryset=Product.objects.filter(category_id=category_id)
    )

4. Invalidating dynamic keys

You do not need to manually invalidate category_items_1, category_items_2, etc. When you call sf.refresh('core_product'), SyncForge clears all cache keys registered under that table name automatically.

5. Cache Stampede Protection

The Problem:

If a popular cache key expires or is refreshed, and 500 users request that page at the exact same millisecond, all 500 requests will experience a cache miss and hit your database simultaneously, bringing it down.

The Solution (built-in):

SyncForge uses Python threading.Lock() per cache key with Double-Checked Locking. Only the first thread executes the database query. The other 499 threads wait for the lock, then instantly retrieve the result from the cache populated by the first thread. Result: exactly 1 database call.
