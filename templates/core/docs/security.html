Security Guide

Security & WAF

SyncForge isn't just for syncing. The SDK includes a production-ready Web Application Firewall (WAF) and comprehensive security headers to protect your Django app.

0. Zero-Data Privacy Architecture

SyncForge operates on a strict zero-data model.

Unlike traditional managed caching or database-as-a-service platforms, the SyncForge central backend never ingests, processes, or stores your application payload data. Customer records, financial transactions, and proprietary business logic remain entirely isolated within your own infrastructure.

What We Sync

Only lightweight synchronization metadata is exchanged over the network: table identifiers, invalidation timestamps, cache keys, and API credentials.

What Stays Local

All query results and cached data structures are stored securely in your local environment (e.g., Redis or Django LocMem).

By managing only metadata, SyncForge inherently reduces your application's external attack surface while maintaining an ultra-lightweight, infinitely scalable backend architecture.

Security Note

No software or architecture can guarantee 100% security. SyncForge follows a privacy-first, zero-data design philosophy intended to significantly minimize risk and limit data exposure, rather than eliminate all security threats.

1. Enable the Security Middleware

Add it directly after Django's built-in SecurityMiddleware.

settings.py 
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'syncforge.middleware.SyncForgeSecurityMiddleware',  # ← Add this line
    'django.contrib.sessions.middleware.SessionMiddleware',
    # ...
]

2. Request Inspection (WAF)

The middleware intercepts every incoming request and scans the URL path, query parameters, and POST bodies for malicious payloads. If a threat is detected, it blocks the request immediately with a 403 Forbidden and logs the attempt.

Threat Vector Patterns Blocked
SQL Injection (SQLi) UNION SELECT, OR 1=1, DROP TABLE, --
Cross-Site Scripting (XSS) <script>, javascript:, onerror=
Path Traversal ../, /etc/passwd
Command Injection /bin/sh, wget, curl

3. Security Headers Injected

The middleware automatically adds these headers to every outgoing response to harden the browser.

X-Content-Type-Options: nosniff

Prevents MIME-sniffing attacks (e.g. executing an uploaded image as JavaScript).

X-Frame-Options: DENY

Prevents Clickjacking by blocking your site from being embedded in iframes on other domains.

X-XSS-Protection: 1; mode=block

Enables legacy browser XSS filters (supplementary to CSP).

Referrer-Policy: strict-origin-when-cross-origin

Prevents leaking sensitive URLs (like password reset tokens) in the Referer header.
