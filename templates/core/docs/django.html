1. Installation

terminal 
pip install syncforge

2. Create the client

Create a dedicated file at your project root — import sf from it everywhere.

myproject/sf.py 
import os
from syncforge import SyncForge

# Initialize the SyncForge Engine
sf = SyncForge(
    api_key=os.environ['SYNCFORGE_API_KEY'],
    backend='django_cache',  # Automatically hooks into your existing settings.CACHES
    silent=True,             # Never crash the app if cache is unreachable
    async_mode=False         # Standard synchronous thread pooling for Django
)

Django Connection Backends

backend='django_cache' Recommended for Django

Hijacks Django's native CACHES framework (e.g., Memcached or DatabaseCache) defined in your settings.py. Transparently utilizes your existing infrastructure without spinning up a new Redis instance.

backend='in_memory'

Uses an ultra-fast Python dictionary to store cached queries directly in RAM. Absolute lowest latency, but cache is wiped on server restarts.

backend='redis'

Connects directly to a Redis cluster. Only use this if you want SyncForge to bypass Django's cache framework and manage its own distributed Redis pub-sub invalidation.

3. @sync_model decorator

Apply @sync_model(sf, table_name='tbl_xxx') to any Django model. On every .save() or .delete() — including admin and shell — SyncForge:

  • → Immediately clears the local cache for this table (synchronous)
  • → Notifies the SyncForge server in a background thread (never blocks)
myapp/models.py 
from django.db import models
from myproject.sf import sf
from syncforge.django import sync_model

# Listen to Django's native post_save/post_delete signals
@sync_model(sf, table_name='tbl_12345', sync_mode='event')
class Product(models.Model):
    name   = models.CharField(max_length=200)
    price  = models.DecimalField(max_digits=10, decimal_places=2)
    active = models.BooleanField(default=True)

    class Meta:
        db_table = 'core_product'

Enterprise Decorator Parameters Explained

sync_mode

What it does: Determines when the SyncForge server is notified. Values: 'event' (on every DB write), 'manual' (only via sf.refresh()), 'schedule_1h', etc.

storage_mode

What it does: 'ram_disk' (default) automatically preloads data from the encrypted disk cache into RAM via background threads, providing zero-latency reads while backing up persistently.

encryption / compression

What it does: Applies AES-256 encryption at rest (True by default). Use compression='lz4' for massive querysets to dramatically reduce I/O wait times.

4. Reading Data (Cache-Aside Pattern)

SyncForge uses a smart cache-aside pattern to protect your database. Follow these 3 steps in your views:

  1. 1
    get_table(): Instantly fetches pre-warmed data from local memory. Zero database hits.
  2. 2
    cache_query(): If memory is empty (cache miss), executes the DB query and automatically saves the result back to memory.
  3. 3
    Combine Both: Use a simple if/else block to ensure only the very first user triggers a database query.
myapp/views.py 
from django.shortcuts import render
from myproject.sf import sf
from .models import Product

def product_list(request):
    # 1. Check RAM / Disk cache instantly
    cached_products = sf.get_table('core_product')
    
    if cached_products:
        products = cached_products
    else:
        # 2. Cache Miss: Query DB, SyncForge saves result automatically
        products = sf.cache_query(
            table_name='core_product',
            queryset=Product.objects.all()
        )
        
    return render(request, 'products/list.html', {'products': products})

Enterprise: Stale-While-Revalidate

SyncForge naturally eliminates cache stampedes during sudden traffic spikes. When an object expires during peak load, the very first thread automatically locks the background DB fetch. Every other simultaneous user thread instantaneously receives the stale payload from RAM, utterly bypassing P99 block latency and keeping your app fully responsive.

Enterprise: Intelligent Heat Engine

You never have to manually define max_items again. The engine dynamically profiles active RAM footprint against a safe 256MB boundary, assigning "Heat Scores" to payloads based on hits and recency. If memory peaks, cold objects gracefully flush themselves.

5. Render data in templates

cache_query() returns a standard Python list of Django model instances — identical to list(queryset). All model fields and methods work normally in templates.

templates/products/list.html 
<!-- Total count -->
<p>Total products: {{ count }}</p>

{% for product in products %}
    <div class="product-card">
        <h3>{{ product.name }}</h3>
        <p>Price: ₹{{ product.price }}</p>
        {% if product.active %}
            <span>Active</span>
        {% endif %}
    </div>
{% empty %}
    <p>No active products.</p>
{% endfor %}

6. Manual refresh after writes

When you're not using @sync_model, call sf.refresh() manually after any database write.

views.py — create, update, delete 
from myproject.sf import sf
from .models import Product

# CREATE
def add_product(request):
    Product.objects.create(name=request.POST['name'], price=request.POST['price'])
    sf.refresh('core_product')   # ← always refresh after write
    return redirect('product_list')

# UPDATE
def update_product(request, pk):
    Product.objects.filter(pk=pk).update(price=request.POST['price'])
    sf.refresh('core_product')
    return redirect('product_list')

# BULK
def bulk_activate(request):
    Product.objects.filter(active=False).update(active=True)
    sf.refresh('core_product')   # One call clears all cache keys for this table
    return redirect('product_list')

7. How signals work internally

When Product.save() is called, Django fires post_save. SyncForge intercepts it:

  1. 1. Instantly deletes all cache keys registered for core_product from Django's cache
  2. 2. Spawns a daemon thread → sends POST to /api/v1/sync/core_product/
  3. 3. Server increments version_number atomically and logs a SyncEvent
  4. 4. Next cache_query() call: cache miss → fresh DB query → store → return

The HTTP call never blocks the request/response cycle.

9. Monthly / scheduled data refresh

For data that changes infrequently (e.g. once a month), use timeout=None and rotate the cache_key by time period. This means the cache is only invalidated by your refresh call — never by a clock.

views.py — monthly cache pattern 
import datetime
from myproject.sf import sf

def monthly_report(request):
    now = datetime.date.today()
    # New cache_key each month = automatic rotation
    key = f'report_{now.year}_{now.month}'

    data = sf.cache_query(
        table_name='core_report',
        cache_key=key,
        queryset=Report.objects.filter(active=True),
        timeout=None,   # Never expires — only sf.refresh() clears it
    )
    return render(request, 'reports/monthly.html', {'data': data})
