API Reference

REST API

If you're not using Python, you can integrate SyncForge using raw HTTP requests. All endpoints require the X-API-Key header.

Base URL

https://syncforge.dev/api/v1
POST /sync/{table_name}/

Signal that a table's data has changed. Increments the version number, purges/invalidates cache, and logs a SyncEvent.

Cooldown Rate Limiting: Table refreshes are subject to a **60-second cooldown** per table per project. Rapid calls within this window return an HTTP 429 Too Many Requests response.

Headers

X-API-Key Required Your project API key (sf_live_...)
X-SF-Timestamp Optional Unix timestamp for replay protection
X-SF-Signature Optional HMAC-SHA256 signature of the request

Response (200 OK)

{ "status": "ok", "message": "Sync triggered for table `users`.", "table": "users", "scoped_table_name": "sf_a1b2c3d4_users", "sync_mode": "Event — On INSERT / UPDATE / DELETE", "database_calls_saved": 1042, "cache_version": 42 }

Response (429 Rate Limit Exceeded)

{ "error": "Rate limit exceeded for table refresh.", "detail": "Please wait 45 seconds before refreshing table `users` again.", "retry_after": 45 }

2. Reading Data (Cache-Aside Pattern)

When building your own client without the Python SDK, you should implement the cache-aside pattern locally. Follow these 3 steps:

  1. 1
    Get Local Cache: Instantly fetch pre-warmed data from your local memory. Zero database hits.
  2. 2
    Query DB & Save: If memory is empty (cache miss), execute the DB query and automatically save the result back to memory.
  3. 3
    Combine Both: Use a simple if/else block to ensure only the very first user triggers a database query.

Pseudo-code Implementation

// Pseudo-code for caching logic
if (local_cache.get("product")) {
    // 1. Cache Hit: Data found in memory! DB query skipped entirely.
    log = local_cache.get("product");
} else {
    // 2. Cache Miss: Execute DB query and automatically save to memory
    log = database.query("SELECT * FROM products");
    local_cache.set("product", log);
}
// 3. Return data exactly as it came from the DB
return render("home", {"product": log});
GET /health/

Ping the SyncForge server to verify your API key is valid and the server is reachable.

Response (200 OK)

{ "status": "ok", "version": "1.1.0", "auth": "valid" }

Switching to Local (Offline) Mode

Developing against the Live Cloud ensures you see real dashboard analytics. However, if you are flying, working completely offline, or want to avoid spamming your live project with test data, you can seamlessly switch your project to Local Dev Mode.

What happens when you switch?

  • SyncForge instantly stops routing REST requests to the cloud edge.
  • Terminal requests and responses are perfectly mocked and logged directly to your standard output.
