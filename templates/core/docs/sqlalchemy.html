Framework Guide

SQLAlchemy Integration

Integrate SyncForge directly with your SQLAlchemy models using the @sync_model decorator. SyncForge automatically hooks into SQLAlchemy's event listeners for cache invalidation.

1. Install

pip install syncforge sqlalchemy

2. Initialise the Client and Models

Use @sync_model on your declarative base models.

from sqlalchemy import Column, Integer, String
from sqlalchemy.orm import declarative_base
from syncforge import SyncForge
from syncforge.sqlalchemy import sync_model
import os

sf = SyncForge(
    api_key=os.environ.get("SYNCFORGE_API_KEY"),
    backend='in_memory'  # Zero infrastructure dependencies
)
Base = declarative_base()

@sync_model(sf, table_name='tbl_12345')
class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True, index=True)
    name = Column(String, index=True)

Enterprise Cache Parameters Explained

The @sync_model decorator accepts several parameters to fine-tune your enterprise caching strategy. Here is what they do and why you should use them:

sync_mode

What it does: Determines when the SyncForge server is notified to update the network.

Values: 'event' (on every database write), 'manual' (only when sf.refresh() is called), 'schedule_5m', 'schedule_1h', etc.

Why use it: Use 'event' for data that needs real-time accuracy. Use 'schedule_1h' for heavy data that is okay being up to an hour stale, reducing server load.

storage_mode

What it does: Defines where the cached data is physically stored on your server.

Values: 'ram_disk' (default), 'disk_only', 'ram_only', 'disabled'.

Why use it: 'ram_disk' is the default. It automatically preloads data from the encrypted disk cache into RAM via background threads, providing zero-latency reads while backing up data persistently and securely.

encryption

What it does: Applies AES-256 encryption to the data stored on disk.

Values: True (default), False.

Why use it: Privacy and compliance. We default to True so that any PII or sensitive cached data is protected at rest automatically.

compression

What it does: Compresses the binary cache payload before storing it.

Values: 'none' (default), 'lz4', 'zstd', 'gzip'.

Why use it: Use 'lz4' or 'zstd' for large JSON objects or massive querysets. It dramatically reduces disk footprint and I/O wait times with very little CPU overhead.

3. Cache-Aside Pattern with get_table()

Always use the 3-step Cache-Aside pattern. Because of the @sync_model decorator, SyncForge automatically invalidates the "users" cache on insert, update, or delete. 

def get_users():
    session = SessionLocal()
    try:
        # 1. Fetch instantly from RAM Cache
        users = sf.get_table("users")
        
        if not users:
            # 2. On cache miss, run the DB query and cache it
            users = sf.cache_query(
                table_name="users",
                queryset=session.query(User).all(),
                timeout=3600
            )
        return users
    finally:
        session.close()

def create_user(name: str):
    session = SessionLocal()
    try:
        new_user = User(name=name)
        session.add(new_user)
        session.commit()
        # 3. Cache is AUTOMATICALLY refreshed here because of @sync_model!
        return new_user
    finally:
        session.close()
